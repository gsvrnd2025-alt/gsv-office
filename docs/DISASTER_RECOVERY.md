# GSV Office — Disaster Recovery & Business Continuity Procedure

This document outlines the disaster recovery objectives, failover procedures, restoration workflows, and system validation steps for the GSV Office platform.

---

## 1. Disaster Recovery Parameters

* **Recovery Point Objective (RPO):** Maximum allowable data loss.
  * **Critical Data (PostgreSQL, Uploads):** 24 hours (based on daily backup cycle).
  * **Non-Critical Data (Redis Cache):** Best effort (re-generates from database).
* **Recovery Time Objective (RTO):** Maximum allowable downtime.
  * **System Outage Restoration:** 30 minutes (once backup archive is available on host).

---

## 2. Backup Archive Structure

All backups generated by `scripts/backup.sh` are compressed `.tar.gz` files containing:
* `database.dump`: Compressed PostgreSQL database dump.
* `redis_cache.rdb`: Redis snapshot cache.
* `minio_data.tar.gz`: Tar archive of MinIO objects (user notes, chat attachments).
* `uploads_data.tar.gz`: Tar archive of main file uploads.
* `configs.tar.gz`: Core configuration files (.env, nginx.conf, mailserver configs).

---

## 3. Disaster Scenarios & Recovery Workflows

### Scenario A: Corruption of Database or Storage Volumes

If files become corrupt or the PostgreSQL database exhibits referential errors, restore from the latest clean daily backup.

**Procedure:**
1. Connect via SSH to the TrueNAS SCALE host.
2. Navigate to the app scripts directory:
   ```bash
   cd /mnt/GSVR_Movies/apps/gsv-office/scripts
   ```
3. Execute the restore script, passing the target backup archive path:
   ```bash
   bash restore.sh /mnt/GSVR_Movies/apps/gsv-office-backups/daily/gsv_office_backup_YYYYMMDD_HHMMSS.tar.gz
   ```
4. Follow the prompt to confirm overwrite. The script will automatically stop applications, clear data volumes, recreate the database schema, import the backup, and start the app services.

---

### Scenario B: Complete TrueNAS Host Failure (Disaster Failover)

In the event of a physical hardware crash where the TrueNAS SCALE host is unrecoverable, follow these steps to restore the platform on new hardware.

#### Step 1: Initialize New OS & Setup ZFS Pools
1. Install TrueNAS SCALE on the replacement hardware.
2. Recreate your storage pools using the ZFS import tool or configuration wizard. Ensure you have the datasets active at `/mnt/GSVR_Movies/` and `/mnt/tank/`.

#### Step 2: Retrieve Backup Archives
1. Locate your offsite backups or retrieve them from the replica disk.
2. Copy the latest backup archive to the temporary folder on the new server:
   ```bash
   mkdir -p /mnt/GSVR_Movies/apps/gsv-office
   scp user@backup-server:/backups/gsv_office_backup_latest.tar.gz /tmp/
   ```

#### Step 3: Clone Codebase and Seeding Setup
1. Clone the project code repository on the new host:
   ```bash
   git clone https://github.com/gsvrnd2025-alt/gsv-office.git /mnt/GSVR_Movies/apps/gsv-office
   ```
2. Navigate to the directory:
   ```bash
   cd /mnt/GSVR_Movies/apps/gsv-office
   ```
3. Initialize the basic folder structures and configurations:
   ```bash
   mkdir -p database nginx/conf.d logs plugins uploads db redis minio downloads mailserver/config
   ```

#### Step 4: Run Initial App Creation
1. Go to **Apps** -> **Discover Apps** -> **Custom App** in the TrueNAS Web UI.
2. Install the app under the name `gsv-office`.
3. Provide the basic compose parameters mapped to the ZFS pools (see [Deployment Guide](DEPLOYMENT_GUIDE.md) for configurations).

#### Step 5: Execute Restore Script
1. Run the restore utility to override the initial install database with the production archive:
   ```bash
   bash /mnt/GSVR_Movies/apps/gsv-office/scripts/restore.sh /tmp/gsv_office_backup_latest.tar.gz
   ```
2. Once the script finishes, the app services will restart and synchronize with the active database state.

## 4. Post-Restoration Checklist

After any restore operation, run the following verification steps:

* **[ ] Service Accessibility:** Open `http://192.168.0.177:8080` in the browser and confirm the login screen loads.
* **[ ] Database Health:** Log in with `admin@gsv.local` and go to **Admin Settings** -> **Server Status**. Verify database size is populated.
* **[ ] File Upload System:** Upload a test document in the Files feature and verify it uploads and generates a preview.
* **[ ] Real-time Socket Connection:** Open the Chat page in two browser sessions and verify messages deliver in real-time without latency.
* **[ ] MinIO Bucket Storage:** Inspect objects and confirm previous chat media attachments are readable.
